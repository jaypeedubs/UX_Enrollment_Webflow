Design — Section 1 of 5: Screens & Navigation

The full inventory of screens in the admin panel and how they connect. Does this match your mental model?

Pages (Astro routes)

Auth
/admin/login
Supabase magic link or email+password for users with role: "admin" in their JWT. Redirects to /admin on success.
Dashboard
/admin
Summary stats: total unreviewed, counts by course, applications needing action (info requested, awaiting review). Quick-links into filtered list views.
Main Panel — Split-Pane ★
/admin/applications
One screen, two panes. All the requested list views are filter states of this single route — no separate pages needed.
All unread (submitted)
By course → submitted
By course → all
By course → waitlisted
By course → accepted
By course → paid
By course → enrolled
Deep Link
/admin/applications/:id
Shareable URL for a specific application. Renders split-pane with that record pre-selected in the detail panel. Useful for bookmarking or sharing between reviewers.
Suggested Addition
/admin/programs (future)
Out of scope for now — Retool handles this. Placeholder route so the nav link exists and can be wired up later.

Suggested screens from codebase review (not in original brief)

Request More Info
Action already in backend. Sends applicant a notification asking them to respond. Needs a compose-message dialog in the panel.
Event Timeline
application_events table already logs everything. Show audit trail in the detail panel: submitted → in_review → accepted, with timestamps and who acted.
CV Inline Viewer
Generate a signed Supabase storage URL and render it in an <iframe> inside the detail panel. No download required to review a CV.
